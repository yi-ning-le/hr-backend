# Agent Configuration: HR Backend

This file defines the operational context, required skills, and development standards for the HR Backend project.

## Project Identity

**Name:** HR Management System - Backend API
**Stack:** Go 1.26+, Gin, PostgreSQL (pgx), sqlc
**Description:** Backend service for an HR management system with clean architecture.

## Commands

- **Build API**: `just build` (Outputs to `bin/server`)
- **Run API (Dev)**: `just dev` (with hot module replacement)
- **Run Migrations**: `just migrate` (Runs migrations and triggers hot reload if Air is running)
- **Re-run Specific Migration**: `just remigrate <migration_file.sql>` (Drops tables, removes record, re-applies, triggers hot reload)
- **Run Tests**: `just test`
- **Run Single Test**: `just test-one ./path/to/pkg TestName`
- **Generate SQL**: `just generate` (Run after modifying SQL in `migrations/` or `internal/repository/query/`)
- **Install Tools & Git Hooks**: `just setup`
- **Lint**: `just lint`

## Skill Matrix

The following specialized skills are relevant to this project and should be utilized to ensure high-quality engineering:

| Skill                 | Purpose       | Context                                                                                                            |
| --------------------- | ------------- | ------------------------------------------------------------------------------------------------------------------ |
| **tdd-first-codegen** | **MANDATORY** | Strictly enforces Test-Driven Development. Tests must be written and fail _before_ implementation code is written. |

## Project Structure

- **Entry Points**:
  - `main.go`: Primary API server entry point. Connects to DB, initializes repositories/services/handlers, and starts Gin server.
  - `cmd/migrate/main.go`: Custom database migration tool.
- **Internal Structure** (`internal/`):
  - `config/`: Configuration loading via environment variables.
  - `handler/`: HTTP handlers (Gin controllers).
  - `middleware/`: HTTP middleware (Auth, CORS, Role-based access).
  - `model/`: Domain models and shared structs.
  - `repository/`: Database access layer. _Auto-generated by sqlc_ from `internal/repository/query/*.sql`.
  - `service/`: Business logic layer.
- **Public Packages** (`pkg/`):
  - `database/`: Database connection logic (pgx pool).
  - `logger/`: Logging utilities.
- **Data Layer**:
  - `migrations/`: SQL schema definitions.
  - `internal/repository/query/`: SQL queries used by `sqlc` to generate Go code.
  - `sqlc.yaml`: Configuration for `sqlc`.

## Architecture Layers

- **Handler**: HTTP transport concerns only (req binding, validation, response marshaling).
- **Service**: Business logic, transactions, coordinating multiple repository calls.
- **Repository**: Pure CRUD operations (generated by sqlc).

## Development Conventions

### Database Access

- **Tool:** Use `sqlc` for all database interactions.
- **Workflow:**
  1. Modify `migrations/*.sql` for schema changes.
  2. Add/Edit queries in `internal/repository/query/*.sql`.
  3. Run `just generate`.
  4. Use generated methods in `internal/repository/`.
- **Prohibition:** Do not write dynamic SQL or raw queries inside Go handlers/services.

### Dependency Injection

- Constructors (e.g., `NewService`, `NewHandler`) must accept interfaces for dependencies to facilitate mocking.
- Dependencies (DB pool, Repositories, Services) are manually wired in `main.go` and injected into constructors.

### Configuration

- Managed via environment variables (defaults in `internal/config/config.go`).

### Error Handling

- Standard Go error handling patterns.

### Testing

- Use `testify` for assertions.

## Operational Mandates

### Test-Driven Development (TDD)

- **Rule:** No implementation code without a failing test.
- **Workflow:**
  1. Create/Modify `_test.go` file.
  2. Define the test case (input, expected output, mock expectations).
  3. Run test -> Verify Failure (Red).
  4. Implement minimum code to pass (Green).
  5. Refactor.

### Database Migrations

- Run `just migrate` to apply pending migrations.
- Run `just remigrate <file.sql>` to re-run a specific migration.
- Both commands trigger hot reload if Air is running.
