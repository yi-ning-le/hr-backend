# Agent Configuration: HR Backend

This file defines the operational context, required skills, and development standards for the HR Backend project.

## Project Identity
**Name:** HR Management System - Backend API
**Stack:** Go 1.25+, Gin, PostgreSQL (pgx), sqlc

## Skill Matrix
The following specialized skills are relevant to this project and should be utilized to ensure high-quality engineering:

| Skill | Purpose | Context |
|-------|---------|---------|
| **tdd-first-codegen** | **MANDATORY** | Strictly enforces Test-Driven Development. Tests must be written and fail *before* implementation code is written. |
| **golang-pro** | Architecture & Idioms | Use for concurrency patterns, error handling, and standard Go project structure adherence. |
| **postgres-pro** | Database Optimization | Use for schema design, indexing strategies, and advanced PostgreSQL features (JSONB, etc.). |
| **api-designer** | API Contracts | Use when designing new endpoints to ensure consistent RESTful conventions and resource modeling. |
| **test-master** | Testing Strategy | Complements TDD by ensuring comprehensive coverage (unit, integration) and proper mocking techniques. |

## Operational Mandates

### 1. Test-Driven Development (TDD)
*   **Rule:** No implementation code without a failing test.
*   **Workflow:**
    1.  Create/Modify `_test.go` file.
    2.  Define the test case (input, expected output, mock expectations).
    3.  Run test -> Verify Failure (Red).
    4.  Implement minimum code to pass (Green).
    5.  Refactor.

### 2. Database Interactions
*   **Tool:** Use `sqlc` for all database interactions.
*   **Workflow:**
    1.  Write raw SQL in `internal/repository/query/*.sql`.
    2.  Run `sqlc generate`.
    3.  Use generated Go code in services.
*   **Prohibition:** Do not write dynamic SQL or raw queries inside Go handlers/services.

### 3. Architecture Layers
*   **Handler:** HTTP transport concerns only (req binding, validation, response marshaling).
*   **Service:** Business logic, transactions, coordinating multiple repository calls.
*   **Repository:** pure CRUD operations (generated by sqlc).

### 4. Dependency Injection
*   Constructors (e.g., `NewService`, `NewHandler`) must accept interfaces for dependencies to facilitate mocking.
